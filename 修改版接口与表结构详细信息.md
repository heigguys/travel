本文档以当前项目源码为唯一事实来源整理，覆盖后端 Controller、DTO、Service、Mapper、`schema.sql`、`application.properties` 以及前端 `app.js` 中实际使用的接口。

前端默认按照当前页面协议和主机拼接后端地址：

```javascript
const API_BASE =
    window.API_BASE ||
    `${window.location.protocol}//${window.location.hostname || "localhost"}:8080/api`;
```

所有前端 API 请求均携带 Cookie：

```javascript
fetch(API_BASE + path, {
    credentials: "include"
});
```

### 1.3 数据库初始化

当前配置：

```properties
spring.sql.init.mode=never
spring.sql.init.encoding=UTF-8
```

后端启动时不会自动执行 `schema.sql` 和 `data.sql`，因此重启服务不会重建数据表或清空咨询消息。

全新数据库需手动按顺序执行：

1. `back-end/src/main/resources/schema.sql`
2. `back-end/src/main/resources/data.sql`

`schema.sql` 用于创建表。若脚本版本包含删除表语句，只能对可清空的数据库执行。`data.sql` 用于写入演示用户、旅行计划、申请和随行人数据。

## 2. 通用规则

### 2.1 用户角色

数据库和 Java 模型使用整数角色：

| 值 | 常量 | 说明 |
| --- | --- | --- |
| `0` | `User.ROLE_ADMIN` | 管理员 |
| `1` | `User.ROLE_USER` | 普通员工 |

主要权限：

- 管理员可以查看全部旅行计划，并按公开状态筛选。
- 管理员可以新增、编辑未公开计划、删除计划和查看所有员工咨询。
- 普通员工只能查看已公开计划。
- 普通员工只能操作自己的申请和随行人。
- 普通员工只能查看自己参与的咨询会话。

### 2.2 登录校验

除 `POST /api/auth/login` 外，当前所有业务接口都要求有效 Session。

`AuthService.currentUser` 的校验流程：

1. 从 Session 读取 `USER_ID`。
2. 未找到时抛出 `请先登录`。
3. 按 ID 查询启用用户。
4. 用户不存在或已禁用时使 Session 失效，并抛出 `登录状态已失效`。

### 2.3 统一 JSON 响应

除 PDF 和附件流接口外，接口统一返回：

```json
{
  "success": true,
  "message": "ok",
  "data": {}
}
```

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `success` | `boolean` | 是否成功 |
| `message` | `string` | 成功提示或错误原因 |
| `data` | `T` / `null` | 业务数据 |

成功且带数据时使用：

```json
{
  "success": true,
  "message": "ok",
  "data": {}
}
```

仅返回成功消息时使用：

```json
{
  "success": true,
  "message": "操作成功",
  "data": null
}
```

## 3. 旅行计划状态

### 3.1 状态值

| 值 | 状态 | 计算条件 |
| --- | --- | --- |
| `0` | 可申请 | 启程日在今天之后，申请总人数小于定员 |
| `1` | 已成团 | 启程日在今天之后，申请总人数大于或等于定员 |
| `2` | 进行中 | 启程日小于或等于今天、返回日大于或等于今天，且人数达到定员 |
| `3` | 已结束 | 返回日早于今天，且人数达到定员 |
| `4` | 未成团 | 已到启程日或已过返回日，但人数未达到定员 |

申请总人数只统计 `applications.status=0` 的有效申请。

### 3.2 刷新时机

- 用户登录成功时批量刷新全部计划状态。
- 每天凌晨 `00:00` 定时批量刷新。
- 申请或取消申请后刷新对应计划状态。
- 管理员更新计划后批量刷新全部计划状态。

### 3.3 申请限制

- 状态 `2`、`3`、`4` 不允许申请，后端返回 `当前计划状态不可申请`。
- 状态 `0`、`1` 当前仍允许进入申请服务。
- 新申请人数加上现有有效申请人数不得超过定员。

## 5. 接口详细信息

### 5.1 认证接口

#### 5.1.1 登录

| 项目 | 内容 |
| --- | --- |
| Method | `POST` |
| URL | `/api/auth/login` |
| 权限 | 无需登录 |
| Content-Type | `application/json` |
| Controller | `AuthController.login` |

请求体：

| 字段 | 类型 | 必填 | 校验 |
| --- | --- | --- | --- |
| `employeeNo` | `string` | 是 | `@NotBlank`，数据库按大小写敏感方式查询 |
| `password` | `string` | 是 | `@NotBlank`，后端计算 MD5 后比对 |

```json
{
  "employeeNo": "S0765",
  "password": "123456"
}
```

返回 `User`，其中 `passwordMd5` 被清空。

业务规则：

- 用户必须存在且 `enabled=true`。
- 登录成功后 Session 写入 `USER_ID`。
- 登录成功后刷新全部旅行计划状态。

主要错误：

- `该用户不存在`
- `密码错误`

#### 5.1.2 退出登录

| 项目 | 内容 |
| --- | --- |
| Method | `POST` |
| URL | `/api/auth/logout` |
| 权限 | 当前代码不主动读取用户，但需持有目标 Session 才有实际退出意义 |
| Controller | `AuthController.logout` |

行为：调用 `session.invalidate()`。

返回：

```json
{
  "success": true,
  "message": "已退出登录",
  "data": null
}
```

#### 5.1.3 获取当前用户

| 项目 | 内容 |
| --- | --- |
| Method | `GET` |
| URL | `/api/auth/me` |
| 权限 | 已登录 |
| 返回 | `User` |

返回前会清空 `passwordMd5`。

#### 5.1.4 修改密码

| 项目 | 内容 |
| --- | --- |
| Method | `POST` |
| URL | `/api/auth/password` |
| 权限 | 已登录 |
| Content-Type | `application/json` |

请求体：

| 字段 | 类型 | 必填 | 校验 |
| --- | --- | --- | --- |
| `oldPassword` | `string` | 是 | `@NotBlank` |
| `newPassword` | `string` | 是 | `@NotBlank` |
| `confirmPassword` | `string` | 是 | `@NotBlank` |

业务规则：

- 原密码 MD5 必须与数据库一致。
- 新密码和确认密码必须一致。
- 新密码不能和旧密码相同。
- 数据库存储新密码的 32 位 MD5 字符串。

主要错误：

- `原密码错误`
- `两次输入的新密码不一致`
- `新密码不能和旧密码相同`

### 5.2 旅行计划接口

#### 5.2.1 查询旅行计划列表

| 项目 | 内容 |
| --- | --- |
| Method | `GET` |
| URL | `/api/plans` |
| 权限 | 已登录 |
| 返回 | `TravelPlan[]` |

Query 参数：

| 参数 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `keyword` | `string` | 否 | 模糊匹配目的地或计划编号 |
| `status` | `integer` | 否 | `0` 至 `4` |
| `published` | `boolean` | 否 | 仅管理员查询时生效 |
| `sort` | `string` | 否 | 排序字段 |
| `sortDir` | `string` | 否 | 只有 `asc` 使用升序，其他值均按降序 |

允许的 `sort`：

- `planNo`
- `destination`
- `startDate`
- `price`
- `capacity`
- `applicantTotal`
- `myApplicantCount`

未传或不支持的排序字段按 `created_at` 排序。

角色规则：

- 普通员工查询条件自动追加 `published=true`。
- 管理员可查看所有计划，并使用 `published` 筛选。

派生字段：

| 字段 | 计算方式 |
| --- | --- |
| `applicantTotal` | 该计划全部有效申请的 `applicant_count` 总和 |
| `myApplicantCount` | 当前用户在该计划下的有效申请人数 |
| `hasUnreadConsultation` | 当前用户是否存在该计划的未读咨询 |

| --- | --- |
| Method | `GET` |
| URL | `/api/plans/{id}` |
| 权限 | 已登录 |
| Path 参数 | `id`：计划 ID |
| 返回 | `TravelPlan` |

普通员工不能查看未公开计划。

#### 5.2.3 新增计划

| 项目 | 内容 |
| --- | --- |
| Method | `POST` |
| URL | `/api/plans` |
| 权限 | 管理员 |
| Content-Type | `multipart/form-data` |
| 返回 | 新增后的 `TravelPlan` |

表单字段：

| 字段 | 类型 | 必填 | 实际业务校验 |
| --- | --- | --- | --- |
| `destination` | `string` | 是 | 非空，最多 10 个 Java 字符 |
| `startDate` | `date` | 是 | ISO 日期，新增时必须晚于今天 |
| `endDate` | `date` | 是 | 不得早于 `startDate` |
| `price` | `decimal` | 是 | `0` 至 `10000` |
| `capacity` | `integer` | 是 | `10` 至 `50` |
| `published` | `boolean` | 否 | 只有严格为 `true` 时保存为公开 |
| `file` | PDF | 否 | 文件名必须以 `.pdf` 结尾 |

计划编号生成规则：

```text
TP + yyyyMMdd + 三位流水号
```

例如：`TP20260624001`。

新计划无申请人，初始状态按日期计算：

- 未开始：`0`
- 已开始或已结束：`4`

主要错误：

- `只有管理员可以执行该操作`
- `请填写所有必填项`
- `目的地不能超过10个字符`
- `价格不能为负数`
- `单人价格上限为 10000 元`
- `定员数不能少于 10 人`
- `定员数不能超过 50 人`
- `返回日不能早于启程日`
- `启程日最早可以选择明天`
- `附件只能上传 PDF 文件`

#### 5.2.6 直接删除计划

| 项目 | 内容 |
| --- | --- |

| 权限 | 管理员 |

#### 5.2.7 邮件通知后删除

| 项目 | 内容 |
| --- | --- |
| Method | `POST` |
| URL | `/api/plans/{id}/notify-cancel-and-delete` |
| 权限 | 管理员 |

业务流程：

1. 查询计划和有效申请人。
2. 没有申请人时直接删除。
3. 有申请人时向每个存在邮箱的申请人发送取消通知。
4. 全部发送成功后删除申请记录和计划。
5. 任一邮件发送失败时抛出异常，事务回滚数据库删除。

邮件包含计划编号、目的地、启程日和返回日。

主要错误：

- `邮件服务未配置，请先配置 spring.mail.host`
- `邮件通知发送失败：{邮箱}`

### 5.3 申请与随行人接口

#### 5.3.1 申请旅行计划

| 项目 | 内容 |
| --- | --- |
| Method | `POST` |
| URL | `/api/plans/{planId}/apply` |
| 权限 | 已登录 |
| Content-Type | `application/json` |
| 返回 | 新增或更新后的 `Application` |

请求体：

| 字段 | 类型 | 必填 | 校验 |
| --- | --- | --- | --- |
| `applicantCount` | `integer` | 是 | `@NotNull`、最小值 `1` |
| `optionText` | `string` | 否 | Service 未附加长度校验，数据库列上限 500 |

业务规则：

- 计划必须存在并已公开。
- 状态 `2`、`3`、`4` 不可申请。
- 变更后的有效申请总人数不得超过定员。
- 同一用户、同一计划已有有效申请时更新原记录，不新增重复有效申请。
- 新申请状态为 `0`。
- 保存后刷新计划状态。

容量计算：

```text
当前有效总人数 - 本人旧申请人数 + 本次申请人数 <= 定员
```

主要错误：

- `旅行计划不存在或未公开`
- `当前计划状态不可申请`
- `申请人数超过定员`

| 权限 | 已登录，只能修改自己的申请 |
| Content-Type | `application/json` |
| 请求体 | `CompanionRequest[]` |

单条随行人字段：

| 字段 | 类型 | 必填 | 校验 |
| --- | --- | --- | --- |
| `name` | `string` | 是 | 2 至 20 位中文、英文、空格或间隔点 |
| `gender` | `string` | 是 | `@NotBlank`，未限制固定枚举 |
| `idCard` | `string` | 是 | 17 位数字 + 1 位数字或 `X/x` |
| `bedNeeded` | `boolean` | 否 | 只有严格为 `true` 时保存为 `true` |

请求示例：

```json
[
  {
    "name": "张三",
    "gender": "男",
    "idCard": "110101199001011234",
    "bedNeeded": true
  }
]
```

业务规则：

- 数组长度不能超过申请人数。
- 保存方式为先删除该申请全部旧随行人，再逐条插入新数据。
- 当前实现允许随行人数少于申请人数。

主要错误：

- `申请记录不存在或无权访问`
- `随行人员数量不能超过申请人数`

### 5.4 咨询接口

#### 5.4.1 查询咨询消息

| 项目 | 内容 |

| 权限 | 已登录且可查看计划 |
| 返回 | `Consultation[]` |

Query 参数：

| 参数 | 类型 | 普通员工 | 管理员 |
| --- | --- | --- | --- |
| `participantUserId` | `long` | 忽略，固定为当前用户 ID | 必填，指定咨询员工 |

业务规则：

- 普通员工只能查询自己和管理员围绕该计划的消息。
- 管理员只能选择已经存在咨询记录的员工。
- 消息按 `created_at`、`id` 升序返回。
- 查询成功后记录当前用户的已读时间。

管理员主要错误：

- `请选择咨询员工`
- `请选择已有咨询员工`

#### 5.4.2 查询计划咨询会话

| 项目 | 内容 |
| --- | --- |
| Method | `GET` |
| URL | `/api/plans/{planId}/consultations/sessions` |
| 权限 | 已登录且可查看计划 |
| 返回 | `ConsultationSession[]` |

管理员返回该计划下真实存在的员工会话：

- 员工 ID、员工编号、姓名。
- 最近消息时间。
- 员工发给管理员且晚于管理员已读时间的消息数量。

普通员工返回一个当前用户的虚拟会话对象，`unreadCount=0`，不查询真实会话列表。

#### 5.4.3 发送咨询消息

| 项目 | 内容 |
| --- | --- |
| Method | `POST` |
| URL | `/api/plans/{planId}/consultations` |
| 权限 | 已登录且可查看计划 |
| Content-Type | `application/json` |
| 返回 | 新增的 `Consultation` |

请求体：

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `content` | `string` | 是 | `@NotBlank`，数据库上限 1000 |
| `participantUserId` | `long` | 管理员必填 | 普通员工提交时会被忽略 |

保存字段：

- `userId`：实际发送人。
- `participantUserId`：会话所属普通员工。
- `senderRole`：发送人角色。
- `status`：固定写入 `OPEN`。

发送成功后发送方对应会话被标记为已读。

#### 5.4.4 关闭咨询会话

| 项目 | 内容 |
| --- | --- |
| Method | `POST` |
| URL | `/api/plans/{planId}/consultations/close` |
| 权限 | 已登录且可查看计划 |

当前实现按以下条件更新：

```sql
where plan_id = #{planId}
  and participant_user_id = #{userId}
```

因此普通员工可关闭自己的会话。管理员调用时使用管理员自身 ID 作为 `participant_user_id`，通常不会匹配普通员工会话。

更新结果：匹配消息的 `status` 统一改为 `CLOSED`。

#### 5.4.5 查询咨询总览

| 项目 | 内容 |
| --- | --- |
| Method | `GET` |
| URL | `/api/consultations/overview` |
| 权限 | 已登录 |
| 返回 | `ConsultationOverviewResponse` |

返回结构：

```json
{
  "unreadTotal": 2,
  "employees": [
    {
      "participantUserId": 2,
      "employeeNo": "S0765",
      "userName": "唐笑松",
      "unreadCount": 2,
      "plans": [
        {
          "planId": 1,
          "planNo": "TP20260615001",
          "destination": "敦煌莫高窟",
          "latestCreatedAt": "2026-06-24T12:47:00",
          "unreadCount": 2
        }
      ]
    }
  ]
}
```

管理员未读规则：

- 只统计 `sender_role=1` 的员工消息。
- 消息时间晚于管理员对该员工、该计划会话的已读时间。
- 管理员已读记录使用 `reader_role=0`、`reader_user_id=0`。

普通员工未读规则：

- 只返回 `participant_user_id=当前用户` 且计划已公开的会话。
- 统计 `user_id != 当前用户` 且晚于本人已读时间的消息。
- 本人已读记录使用 `reader_role=1`、`reader_user_id=当前用户 ID`。

| `findById` | 查询申请 |
| `insert` / `update` | 新增或更新申请 |

### 8.4 CompanionMapper

### 8.5 ConsultationMapper

### 8.6 删除流程说明

由于数据库没有外键级联，删除计划时由 Service 显式执行：

```text
applications.deleteByPlan(planId)
travel_plans.delete(planId)
```

当前删除流程没有显式删除：

- `companions`
- `consultations`
- `consultation_reads`

因此是否产生孤立数据取决于实际数据库是否另有未体现在 `schema.sql` 中的约束。按当前源码和当前 `schema.sql`，这些关联数据不会自动级联删除。

## 11. 当前实现注意事项

1. 密码使用 MD5 摘要，不包含盐值；这是当前源码行为，不代表推荐的生产安全方案。
2. `PlanRequest` 注解允许价格最高 `99999.99`、定员最低 `1`，但 Service 实际限制为价格最高 `10000`、定员 `10` 至 `50`，最终以 Service 规则为准。
3. `schema.sql` 当前没有数据库外键，关联完整性主要由 Service 维护。
4. 已取消申请仍保留随行人记录，查询随行人接口也只校验所有权，不校验申请状态。
5. 关闭咨询接口对管理员会话的匹配方式存在限制，前端当前也未调用该接口。
6. `spring.sql.init.mode=never` 表示修改 `data.sql` 后不会自动更新已有数据库。
7. 本文档描述当前源码实际实现，不承诺未在源码中实现的接口、约束或级联行为。
