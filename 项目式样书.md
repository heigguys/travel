# 公司旅行管理系统 项目式样书

## 1. 文档概要

### 1.1 文档目的

本文档为“公司旅行管理系统”的项目式样书，按功能点整理系统规格。每个功能点统一包含功能概要、使用角色、相关画面、接口、输入输出、业务处理、数据库影响、异常提示和测试项目，便于开发、测试和验收人员按功能逐项确认。

### 1.2 系统概要

公司旅行管理系统是面向公司内部员工的旅行计划管理与申请系统。管理员负责旅行计划维护、PDF 附件上传、申请情况确认和咨询回复；普通员工负责浏览公开旅行计划、提交申请、维护随行人员、查看个人申请、导出 PDF 和发起咨询。

### 1.3 文档范围

| 项目 | 说明 |
| --- | --- |
| 系统名称 | 公司旅行管理系统 |
| 后端工程 | `back-end` |

| 默认后端地址 | `http://localhost:8080` |
| 默认 API 地址 | `http://localhost:8080/api` |

## 4. 功能点规格

### 4.1 登录 / 退出

#### 功能概要

用户通过员工编号和密码登录系统。登录成功后进入旅行计划一览页；退出后清除登录状态并返回登录页。

#### 使用角色与权限

| 角色 | 权限 |
| --- | --- |
| 管理员 | 可登录、退出 |
| 普通员工 | 可登录、退出 |

#### 相关画面

| 画面 | 文件 / 元素 | 说明 |
| --- | --- | --- |
| 登录页 | `front-end/src/main/webapp/index.jsp` | 输入员工编号和密码 |
| 旅行计划一览页 | `front-end/src/main/webapp/plans.jsp` | 登录成功后进入 |
| 退出按钮 | `logoutBtn` | 点击后调用退出接口 |

#### 接口规格

| 接口 | Method | 权限 | Content-Type | 返回 |
| --- | --- | --- | --- | --- |
| `/api/auth/login` | POST | 无需登录 | `application/json` | `ApiResponse<User>` |
| `/api/auth/logout` | POST | 需登录 | 无请求体 | `ApiResponse<Void>` |
| `/api/auth/me` | GET | 需登录 | 无请求体 | `ApiResponse<User>` |

#### 输入项

登录请求 Body：

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `employeeNo` | string | 是 | 员工编号，如 `A001` |
| `password` | string | 是 | 明文密码，如 `123456` |

```json
{
  "employeeNo": "A001",
  "password": "123456"
}
```

#### 输出项

登录成功返回当前用户信息，密码字段不用于前端展示。

| 字段 | 说明 |
| --- | --- |
| `id` | 用户 ID |
| `employeeNo` | 员工编号 |
| `name` | 用户姓名 |
| `email` | 邮箱 |
| `role` | `ADMIN` 或 `USER` |
| `enabled` | 是否启用 |

退出成功返回：

```json
{
  "success": true,
  "message": "已退出登录",
  "data": null
}
```

#### 业务处理

- 后端根据员工编号查询启用状态用户。
- 将输入密码转 MD5 后与 `users.password_md5` 比对。
- 登录成功后将用户 ID 写入 Session 的 `USER_ID`。
- 前端访问登录页时会尝试调用 `/api/auth/me`，若已有登录状态则跳转到旅行计划一览页。
- 退出时调用 `session.invalidate()` 清除会话。

#### 数据库影响

| 表 | 操作 | 说明 |
| --- | --- | --- |
| `users` | SELECT | 根据 `employee_no` 查询用户 |

#### 异常与提示

| 场景 | 提示 |
| --- | --- |
| 员工编号或密码错误 | 员工编号或密码错误 |
| 未登录调用 `/api/auth/me` | 请先登录 |
| 登录用户不存在或被禁用 | 登录状态已失效 |

#### 测试项目

| 编号 | 操作 | 预期 |
| --- | --- | --- |
| AUTH-01 | 使用 `A001 / 123456` 登录 | 登录成功，进入管理员视图 |
| AUTH-02 | 使用 `U001 / 123456` 登录 | 登录成功，进入普通员工视图 |
| AUTH-03 | 输入错误密码登录 | 登录失败并提示错误 |
| AUTH-04 | 登录后刷新页面 | 通过 `/api/auth/me` 保持登录状态 |
| AUTH-05 | 点击退出 | Session 清除，返回登录页 |

### 4.2 修改密码

#### 功能概要

已登录用户可在旅行计划一览页打开修改密码弹窗，输入原密码和新密码完成密码变更。

#### 使用角色与权限

| 角色 | 权限 |
| --- | --- |
| 管理员 | 可修改自己的密码 |
| 普通员工 | 可修改自己的密码 |

#### 相关画面

| 画面 / 元素 | 文件 / ID | 说明 |
| --- | --- | --- |
| 修改密码按钮 | `passwordBtn` | 打开修改密码弹窗 |
| 修改密码弹窗 | `passwordDialog` | 输入原密码和新密码 |
| 修改密码表单 | `passwordForm` | 提交密码修改 |

#### 接口规格

| 接口 | Method | 权限 | Content-Type | 返回 |
| --- | --- | --- | --- | --- |
| `/api/auth/password` | POST | 需登录 | `application/json` | `ApiResponse<Void>` |

#### 输入项

请求 Body：

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `oldPassword` | string | 是 | 当前密码 |
| `newPassword` | string | 是 | 新密码 |

```json
{
  "oldPassword": "123456",
  "newPassword": "654321"
}
```

#### 输出项

修改成功返回：

```json
{
  "success": true,
  "message": "密码修改成功",
  "data": null
}
```

#### 业务处理

- 后端根据 Session 获取当前用户。
- 原密码转 MD5 后与数据库当前密码比对。
- 新密码不能与旧密码相同。
- 新密码转 MD5 后更新 `users.password_md5`。
- 前端新密码输入框设置最小长度为 6。

#### 数据库影响

| 表 | 操作 | 说明 |
| --- | --- | --- |
| `users` | SELECT | 获取当前用户 |
| `users` | UPDATE | 更新 `password_md5` |

#### 异常与提示

| 场景 | 提示 |
| --- | --- |
| 未登录 | 请先登录 |
| 原密码错误 | 原密码错误 |
| 新旧密码相同 | 新密码不能和旧密码相同 |
| 参数为空 | 输入内容不完整或格式不正确 |

#### 测试项目

| 编号 | 操作 | 预期 |
| --- | --- | --- |
| PWD-01 | 输入正确原密码和新密码 | 修改成功 |
| PWD-02 | 输入错误原密码 | 返回原密码错误 |
| PWD-03 | 新密码与旧密码相同 | 返回不能相同 |
| PWD-04 | 未登录调用接口 | 返回请先登录 |

### 4.3 旅行计划查询

#### 功能概要

用户登录后查看旅行计划列表，可按关键字、状态和排序条件筛选。管理员可查看全部计划，普通员工只能查看已公开计划。

#### 使用角色与权限

| 角色 | 权限 |
| --- | --- |
| 管理员 | 查看全部旅行计划，包含未公开计划 |
| 普通员工 | 只查看 `published=true` 的旅行计划 |

#### 相关画面

| 画面 / 元素 | 文件 / ID | 说明 |
| --- | --- | --- |
| 旅行计划一览页 | `plans.jsp` | 展示旅行计划表格 |
| 关键字输入框 | `keywordInput` | 搜索目的地或计划编号 |
| 状态筛选 | `statusFilter` | 未开始、进行中、已结束 |
| 排序选择 | `sortSelect` | 日期、价格、定员 |
| 搜索按钮 | `searchBtn` | 触发重新查询 |

#### 接口规格

| 接口 | Method | 权限 | Content-Type | 返回 |
| --- | --- | --- | --- | --- |
| `/api/plans` | GET | 需登录 | Query String | `ApiResponse<List<TravelPlan>>` |

#### 输入项

`GET /api/plans` 查询参数：

| 参数 | 类型 | 必填 | 说明 |

- 普通员工列表不展示公开状态列、添加按钮、编辑按钮、删除按钮。
- SQL 查询中计算有效申请总人数和当前用户申请人数。
- 默认排序按创建或数据默认顺序；指定排序时按日期、价格或定员排序。

### 4.4 旅行计划新建 / 修改

#### 功能概要

管理员可新建旅行计划，也可编辑已有计划。旅行计划包含目的地、日期、价格、定员、公开状态和 PDF 附件。

#### 使用角色与权限

| 角色 | 权限 |
| --- | --- |
| 管理员 | 可新建和修改旅行计划 |
| 普通员工 | 不可新建或修改旅行计划 |

#### 相关画面

| 画面 / 元素 | 文件 / ID | 说明 |
| --- | --- | --- |
| 添加旅游计划按钮 | `newPlanBtn` | 管理员可见 |
| 旅行计划弹窗 | `planDialog` | 新建和编辑共用 |
| 旅行计划表单 | `planForm` | 提交计划数据 |
| 编辑按钮 | `data-action="edit"` | 打开并填充计划内容 |

#### 接口规格

| 接口 | Method | 权限 | Content-Type | 返回 |
| --- | --- | --- | --- | --- |
| `/api/plans` | POST | 管理员 | `multipart/form-data` | `ApiResponse<TravelPlan>` |
| `/api/plans/{id}` | POST | 管理员 | `multipart/form-data` | `ApiResponse<TravelPlan>` |

#### 输入项

表单字段：

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `destination` | string | 是 | 目的地 |
| `startDate` | date | 是 | 启程日 |
| `endDate` | date | 是 | 返回日 |
| `price` | decimal | 是 | 价格，不能小于 0 |
| `capacity` | integer | 是 | 定员数，至少 1 |
| `published` | boolean | 否 | 是否公开 |
| `file` | file | 否 | PDF 附件 |

#### 输出项

返回保存后的旅行计划对象，包含自动生成的 `id`、`planNo`、`status`、`filePath`、`fileName` 等字段。

#### 业务处理

- 后端先校验当前用户是否为管理员。
- 必填项为空时返回业务异常。
- 返回日不能早于启程日。
- 新建时自动生成旅行计划编号。
- 新建计划默认状态为“未开始”。
- 附件不为空时保存到本地 `storage` 目录。
- 附件只允许 PDF 文件。
- 修改时若上传新附件，则更新文件路径和文件名；否则保留原附件信息。

#### 数据库影响

| 表 | 操作 | 说明 |
| --- | --- | --- |
| `travel_plans` | INSERT | 新建旅行计划 |
| `travel_plans` | UPDATE | 修改旅行计划 |

#### 异常与提示

| 场景 | 提示 |
| --- | --- |
| 未登录 | 请先登录 |
| 非管理员操作 | 只有管理员可以执行该操作 |
| 必填项为空 | 请填写所有必填项 |
| 返回日早于启程日 | 返回日不能早于启程日 |
| 上传非 PDF 文件 | 附件只能上传 PDF 文件 |
| 文件名不合法 | 文件名不合法 |

## 6. 部署与运行

### 6.1 后端启动

1. 启动 MySQL。
2. 确认数据库连接配置：
   - 数据库名：`db_travel`
   - 用户名：`root`
   - 密码：`1234`
3. 进入后端目录并启动：

```cmd
cd back-end
mvnw.cmd spring-boot:run
```

后端启动地址：`http://localhost:8080`。

### 6.2 前端打包

```cmd
cd front-end
mvnw.cmd package -DskipTests
```

WAR 包路径：

```text
front-end/target/front-end-1.0-SNAPSHOT.war
```

### 6.3 前端部署

1. 修改 Tomcat 端口为 `8081`，避免与后端 `8080` 冲突。
2. 删除 Tomcat 默认 `webapps/ROOT`。
3. 将 WAR 包复制为 `webapps/ROOT.war`。
4. 启动 Tomcat。
5. 访问 `http://localhost:8081`。
